Inside the GCE VM — the network

5-CSR iBGP chain, agent on a data-plane veth

Why eth2 matters

A direct veth from dd-agent eth2 to CSR1 Gi4 on 10.99.0.0/30 bypasses vrnetlab's SLiRP-NAT'd Mgmt-intf VRF — without it, ICMP probes to 10.100.X.1 get NAT'd back to the bridge.

Routing

FRR eBGP (AS 65001) peers with CSR1 (AS 65000). CSR1↔CSR5 are full iBGP with belt-and-suspenders static routes for fast cold-boot convergence.

SNMP

All five CSRs polled via SNMPv3 (auth: SHA, priv: AES) using the bundled cisco-csr1000v profile through the core loader.

Concept

How NDM works — in 5 lines

1

Define devices

Agent reads instances.yaml (or autodiscovers a CIDR).

2

Match a profile

For each device, agent issues GetRequest sysObjectID.0 and matches against a vendor profile YAML (200+ bundled — Cisco, Juniper, F5, Palo Alto …).

3

Walk OIDs

Profile declares which OIDs to walk. Agent walks them every min_collection_interval (default 15s).

SE talking point

"For your fleet, we already have profiles for the top 200 vendors. For the remaining 5%, drop a YAML in profiles/ and the core loader picks it up — no agent rebuild."

4

Send to Datadog

Metrics, tags, metadata, interface state, BGP table, CDP neighbors all flow into the same backend as APM / Logs / Infra.

5

View at /devices

Fleet inventory at /devices with per-device drill-downs, topology map, geomap, monitors.

Bundled features
SNMP polling SNMP traps NetFlow CDP / LLDP topology Subnet autodiscovery Custom profiles
Concept

How NetworkPath works — in 5 lines

1

Define destinations

Agent reads network_path.d/conf.yaml — list of hostnames/IPs to reach.

2

Run traceroute

For each, agent's system-probe sends N parallel TCP / UDP / ICMP traceroutes per check run.

3

Capture every hop

Each probe records every hop's IP, RTT, and reachability as a structured event.

4

Enrich + send

Hops are enriched with reverse-DNS / NDM device match, and time-series stored in Datadog.

5

Query continuously

"Avg RTT to hop 3 over last 24h?", "Reachability dropped <99%?", "Did the path change?" — answered at /network/path.

Why it's useful

Answers the "which hop is slow?" question that historically required a war-room call to the network team and a manual traceroute from a jumphost.

Sample query — RTT per hop on a path
avg:datadog.network_path.path.hop_rtt {path_name:agent-to-csr5-cnx-endpoint} by {hop_index, hop_ip_address}
Sample query — path reachability over time
avg:datadog.network_path.path.reachable {path_type:data-plane} by {destination_service}
TCP UDP ICMP 3 traceroutes / interval e2e probes (50 / interval) reverse-DNS enrichment NDM device match
Concept

NDM and NetworkPath — complementary, same backend

Customer question Product that answers it
"Is the device healthy?" — CPU, memory, fans, interface counters, BGP state NDM SNMP polling, per-device profile
"Is the path through these devices healthy?" — RTT, loss, hop changes NetworkPath traceroute-as-metric

"Did routing converge after that change?" NDM BGP table + NetworkPath path-change events
"Did my app's slowdown coincide with a network issue?" Cross-product correlation: APM + NDM + NetworkPath in one dashboard

Inject 200 ms on CSR2 → hop 2 lights up

Observed in this lab

Pre-injection — agent → csr5 (5 hops):

hop 1: 10.99.0.1 1.5 ms hop 2: 10.0.12.2 1.7 ms hop 3: 10.0.23.2 2.0 ms hop 4: 10.0.34.2 2.5 ms hop 5: 10.100.5.1 3.0 ms

Post-injection (same path):

hop 1: 10.99.0.1 1.5 ms hop 2: 10.0.12.2 170+ ms 🔴 hop 3: 10.0.23.2 470 ms hop 4: 10.0.34.2 360 ms hop 5: 10.100.5.1 370 ms

Latency isolated at hop 2 (CSR2 Gi2). Hop 1 stays clean.

Recommended Datadog monitor (Terraform)
resource "datadog_monitor" "np_hop_rtt_high" { name = "[NetworkPath] Hop RTT > 100ms on {{path_name.name}} hop {{hop_index.name}}" type = "metric alert" query = "avg(last_5m):avg:datadog.network_path.path.hop_rtt{path_type:data-plane} by {path_name,hop_index} > 100" # by {hop_index} = the magic — alerts fire PER hop, not per path }
Demo Scenario 2 — Load test against CSR-WAN-TRANSIT

The unlicensed CSR1000v throttles at ~100 Kbps

A bundled loadtest.sh uses hping3 to overload CSR2's data-plane shaper and posts Datadog events at start/end so the test window annotates NetworkPath graphs automatically.

Five severity modes

steady~80 KbpsUnderFlat 0% loss

Demo sequence (~10 min)
# Green baseline loadtest.sh csr-wan-transit steady 120 # Subtle brownout — most production-like loadtest.sh csr-wan-transit trickle 120 # Clear degradation loadtest.sh csr-wan-transit burst 90 # Red alert loadtest.sh csr-wan-transit overload 60
Auto-annotated

Each run posts LoadTest START + LoadTest END events tagged scenario:loadtest mode:<mode> target:<target> — they appear as flags on every NetworkPath graph that overlaps the window.

Expected NetworkPath signature

steady — flat ~1-3ms RTT, 0% loss. Baseline showcase.

trickle — sawtooth latency 2-9× baseline, 0% visible loss. Brownout signature.

burst — clear spikes, 10-30% loss. Customer pages on it.

overload — >99% loss for 60s, 1-2 min recovery tail. Total outage.

Standard Datadog Agent config — copy-paste

conf.d/snmp.d/conf.yaml  [setup docs]
instances: - ip_address: 10.10.20.5 snmp_version: 3 user: dduser authProtocol: SHA authKey: <auth-pass> privProtocol: AES privKey: <priv-pass> loader: core # faster, scales further profile: cisco-csr1000v # skip autodetect tags: - "device_namespace:prod-network" - "site:bkk-dc1" - "team:netops"
conf.d/network_path.d/conf.yaml  [setup docs]
instances: - hostname: csr5-cnx-endpoint protocol: ICMP source_service: dd-lab-agent destination_service: CSR5 endpoint tags: - "path_type:data-plane" - "expected_hops:5" max_ttl: 12 traceroute_queries: 3 min_collection_interval: 60
Subnet autodiscovery in datadog.yaml  [scanner docs]
network_devices: namespace: prod-network autodiscovery: enabled: true workers: 10 discovery_interval: 300 configs: - network_address: 10.10.20.0/24 snmp_version: 3 # … same auth/priv as above …
SNMP traps receiver in datadog.yaml  [traps docs]
network_devices: snmp_traps: enabled: true port: 162 bind_host: 0.0.0.0 community_strings: - <ro-community> users: - username: dduser authKey: <auth-pass> privKey: <priv-pass> # Then on each device: # snmp-server host <agent> v3 priv dduser

What survives terraform destroy — by design

✓ Keeps

GCS image-cache bucket (~2 GB qcow2 + tarball) — force_destroy = false by default. The bucket is the reason the next apply takes 12 min, not 25.

✗ Destroys

VM, VPC, subnet, Cloud Router, Cloud NAT, firewalls, IAM binding, boot disk. destroy errors cleanly on the bucket — that's the safety net firing.

⚙ Override

terraform destroy -var image_cache_force_destroy=true wipes the cache too. Use when rotating qcow2 or starting fully fresh.
